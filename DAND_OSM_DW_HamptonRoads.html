Author: Ashley Goodman¶

Map Area: Hampton Roads, Virginia, United States¶

I selected the Hampton Roads area as its where I grew up, but left 20 years ago. I thought I would take the opportunity this project presents to indulge in some nostalgia.

My analysis process¶

Note: at the end of the document is a list of links to the various files and scripts used in inspecting, cleaning and shaping the data. I have linked to them as, due to file size, they would take too long to upload and may be larger than the allowed file submission size

I sampled using the provided code found in the project setup materials: https://goo.gl/INY68X

I elected to examine and clean the data before exporting it into MongoDb. Step 5: A closer look at the problem characters¶

I look online for references to "pcanet" and osm, but was unable to find any. I decided to look at the entry in the original osm (xml) file.

The challenge was finding a text editor that could handle such a large file. I ended up using EMEditor after some trial and error.

It turned out that the problem characters were found on only one entry so I manually edited (deleted) the sections of the entry as they added no information of value to the data set for the purposes of my analyis. The resulting file: "hamptonRoadsVirginiaManEdit2.osm" is included in the link list.

With each stage I expanded the expected list until only problem names were left

Step 7: Manual inspection of some of the problem names¶

I noticed that some of the problem names were one-of's and were more data entered incorrectly (for example suite numbers included with the street name) rather than mispellings, abbreviations etc.. So I decided to take a manual peek.

A manual inspection confirmed this to be the case and so for 14 of the entries I manually edited the osm file rather than write code to correct them as it would take longer to write the code and I was already looking at them.

Were I to look at multiple areas in an analysis instead of just the one I would probably look at writing code to handle the edits programatically, but as I was just inspecting one area a manual solution was more time efficient.

Here are a 5 examples of the 14 one-of's I manually edited

  1. "Pickett's": set(["Pickett's"]) Which became Picketts's Line

  2. '23529': set(['Parker Ave, Norfolk, VA 23529']), Which became Parker Avenue

  3. '9': set(['Crusader Circle #9']) Which became Crusader Circle

  4. ''45th': set(['45th']) Which became a variety of street names depending on the surrounding tag data and a Google map checks

  5. 'Complex': set(['Krishana Complex']), was deleted as its actually an address in India

Of all of them the last one was most interesting as it was actually an address in India. As a guess I would imagine it was a misguided attempt at SEO.

For each edit I took the time to confirm the data using Google Maps and made sure that items like Suite numbers etc. were added as tags when stripped out of the street name if they were not already present.

In total this process took less than 30 minutes.

It also afforded me the opportunity to look more closely at the data and identify a number of other issues including:

  1. Place and Street names in all caps
  2. Multiple zip code tag types (as examined further below)
  3. Multiple city tag types (as examined further below)

After the manual edit I updated my list of expected names and performed a final check of street names with the final problem name output shown below

Step 8: Postal codes¶

As mentioned above there are 3 tag types for post codes

  1. The expected: addr:postcode
  2. and the unexpected tiger:zip_right
  3. and tiger:zip_left

I searched the OSM wiki for more information on the tiger data: https://goo.gl/XQ4Qbb

The data is outdated (imported in 2007) and will not be imported again.

I read further to see how others have handled the zip_right and left and found no consensus

  1. https://goo.gl/9pf3oY
  2. https://goo.gl/UwJePH

I decided to create 3 sets and examine them using the postcodes script

I then examined the post codes sets for differences using the output of the above code and areas that would need cleaning and created a combined set

This resulted in a complete set of postcodes which allowed me to identify problem areas. (shown in the code file linked to at the bottom of this document)

  1. Postcodes with 2 post codes, for example '23325:23523'
  2. Postcodes with the state abbreviation prepended, for example 'VA 23455'
  3. Postcodes with more than 5 digits, for example '23669-4019'

I examined the full data file manually again and noticed that the zip_right and zip_left were almost always identical for the same entry and that they generally never occured in entries that had addr:postcode

I decided that I would take addr:postcode as the primary source for post codes and where it was lacking would use zip_right

Step 9: Look at city names¶

I took a look at city names using the code below.

I decided to look at all 3 tag types for city names

Aside from the addr:city tag there was also the following 2 tags

  1. gnis:county_name
  2. tiger:county

Looking at the output from the "process_city_names" script the above I noticed several issues

  1. Capitalization like newport news instead of Newport News
  2. Appending (city) to the city name for example, Hampton (city)
  3. The final part of the name was missing where the entry was Isle of Wight instead of Isle of Wight County
  4. Multiple city names as one entry like: James City, VA:York, VA
  5. Appending the state to the city name like Newport News, VA
  6. Mispellings like Noroflk instead of Norfolk

From this I was able to determine that city names would need some cleaning as well for mispelling, capitalization, stripping of extra data like state names, extra characters like commas, city type in parens etc..

Some of this I would do through mapping as shown below, but some I would take care of using scripts as it was quicker to write the script then to extend the mapping, for example capitalization.

I also noticed some entries for North Carolina which while close to Virginia were not technically a part of Hampton Roads, also often in the case of the Tiger data multiple cities would be included in one entry. A look at the osm file showed that these were "ways" and as such acceptable, for example an entry for "Camden, NC:Chesapeake, VA:Currituck, NC" was for the Norfolk Southern Railway.

I now had to decide what data to use, when, how and where.

I created the 2 mappings to assist with the data cleaning process streetmapping and citymapping dictionairies (shown in the code file linked to at the bottom of this document)

Step 10: Cleaning and shaping the data¶

I decided that "addr" info would be used as the primary source, where it was lacking I would use "gnis" or "tiger" data.

The issues to address in my cleaning and shaping of the data would be

  1. Correcting capitalization on place names, street names and street types. So names that were all uppercase or all lower case would be changed to upper case for only the first letter of each word in the name, abbreviations and mispellings would be corrected using the mapping dictionairies above, Where multiple place or street names were present only one (the first) would be used
  2. Normalizing post codes to be 5 digits, where multiple post codes were present only the first will be used
  3. Defaulting the "state" field to "Virginia"
  4. Preserving other key/value data like amenities/ type etc

  5. Shaping the data into nested JSON documents with the following structure

    "pos" : [

                 36.8854858,
             -76.2086654
     ],
     "type" : "node",
     "id" : "39678633",
     address: {
             "street": "Some street",
             "city": "Some City",
             "postcode" : "23456"
             "state": "Virginia"

     "created" : {
             "uid" : "1505651",
             "changeset" : "32900513",
             "version" : "2",
             "user" : "CynicalDooDad",
             "timestamp" : "2015-07-27T03:08:42Z"
     }

Note

  1. That the "pos" field will be floats, all other fields will be strings.
  2. The "created" and "address" fields would be nested documents within each JSON document
  3. The "pos"field would be an array as a nested document within each JSON document
  4. A field created to identify the data type as "node" or "way". All other data types, like"relation" would be removed
  5. In the case of ways, a nested document containing all "nd" refs as list
  6. Fields like amenity, building, cuisine etc would be preserved

I used several scripts to clean and shape the data, all of which are shown in the code file linked to at the bottom of this document

I then used the "process_map" script to produce the actual JSON document ready for import into mongodb

Ideas for additional improvements¶

  • An idea that came to me immediately was for a mobile app that semi-automates the process of updating the map using GPS, cross checking with other map apps and available online databases, like Four Square or even the Tiger database. I say semi automate as it would still require user confirmation that the data was correct before submission. This could be handled through a "check-in" process much like Facebook and Google and other apps currently use. It turns out that such an app does already exist "OSMTracker for Android" by Nicolas Guillaumin though it does not have the cross checking feature
To make this idea work really well a significant hurdles will need to be overcome.¶
  • Name recogition: relatively few people have ever heard of Open Street Map. They rely on other services like Google Maps and are comfortable with the service they are using. Open Street Map needs to engage in some form of marketing for brand recognition and user adoption. Mozilla was able to do this with Firefox by encouraging extension and theme develeopment and attracting developers who in turn became "evangelizers" for Firefox. Open Street Map has an open API, but they need to do more to build a "marketplace/infrastructure" system that encourages developers to both develop and "evangalize" Open Street Maps bringing it more into the public eye, see gamification section for further thoughts on this.
Another idea for improvement is adopting standards¶

  • Some standardization of data as it is input into OSM would be beneficial, for example if entering a node location, where applicable, requiring specifying if it is an amenity, a house/apartment building, for leisure etc. and disallowing entries with all caps or all lowercase before allowing the data to be saved would perhaps help avoid further incomplete or dirty (in the data sense) entries.

    The challenge here is that this is somewhat at odds with the Open Street Map ethos of "you are free to tag things how you want". If they want to become a serious player in accurate mapping a standard must be set and accepted by the community and rigorously applied. Wikipedia is a perfect example of this. It's open and anyone can make edits, however there are standards on everything from what is allowed as an entry to how entries are formatted, citations etc. and they are vigorously enforced by the community.

Gamification¶

  • It has been mentioned in other analyses of Open Street map data for this very project, but I believe it is worth mentioning again with a more specific/literal interpretation:

    Partnering with other games/apps that already make use of map data or creating new apps that do could have multiple benefits.

    1. If those apps make use of another map source as primary, for example Google Maps, the data being entered into Gogle Maps by the app could be consideed a quality source and duplicated in OSM. For example Fitbit's use of maps for tracking running exercices

    2. Apps like Pokemon Go have shown how an app can motivate people to travel around their local areas, Partnering with such an app or creating one that can produce similar results could help collect/correct data in OSM. For example perhaps an app similar to Pokemon Go could have spawn points set not by arriving at a location but rather by entering a location in that does not yet exist in OSM or editing/correcting one that does.

      Businesses could also be encouraged to enter their data in as a spawn point to encourage mjore foot traffic.

      The completeness and type of data entered could determine the spawned creatures type/ strength/ stats and so on.

      There is risk that the system could be gamed with fake entries, but a wiki style approach requiring multiple independent verfication of the entered data combined with in game penalties if entered data is later proven falseified or external confimation using another source as canonical, like Google Maps, could help mitigate the risk.

Idea for further analysis¶

  • Combining population census figures, crime rate figures and cost of living figures with city locations and using that to measure quality of life by the ratios of the number of schools, amenities, police departments, fire departments, leisure areas, grocery stores, malls, shops, churches etc.. by person and by dollar.

    There will be challenges with this as care will need to be taken to be sure that external data like cost of living and crime statistic info for a given locale will cover the same geographic area the Open Street Map of the same locale name does

    This will also require the Open Street Map data to be more complete, standardized and accurate or the quality of life conclusions drawn for an analysis will be mostly inaccurate.

Conclusion¶

In as much as the quality and completeness of the Hampton Roads area Open Street Map data is representative of whole of OSM it is far from complete. It suffers from:

  • Out of date data: for example much of the gnis and tiger data were mass automated imports and there are roads, buildings and even schools in the database that no longer exist
  • Incomplete data: for example Chanello's pizza is easily in the top 5 pizza places, having over a dozen locations, yet has only 1 entry in the OSM data
  • A lack of standardization which can make database queries difficult as the same data type might be recorded in different ways using different tag keys
